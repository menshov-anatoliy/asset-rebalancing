## 1. Что такое HRP и зачем он нужен

**Hierarchical Risk Parity (HRP)** — алгоритм построения инвестиционного портфеля, предложенный Маркосом Лопес де Прадо (Marcos López de Prado) в 2016 году. Его главная идея: использовать **иерархическую структуру корреляций** между активами, чтобы распределить риск равномерно — без необходимости обращать ковариационную матрицу (что является узким местом классической теории Марковица).

### Почему не Марковиц?

Классический метод «оптимального портфеля» Марковица (Mean-Variance Optimization, MVO) страдает рядом системных проблем:

| Проблема | Описание |
|---|---|
| **Нестабильность весов** | Небольшое изменение оценок доходностей приводит к радикальному пересмотру весов |
| **Концентрация риска** | Оптимальный портфель нередко вкладывает почти всё в 1–2 актива |
| **Обратная матрица** | Инверсия ковариационной матрицы численно нестабильна при большом числе активов |
| **Прогноз доходностей** | Требует точных оценок ожидаемых доходностей, которые сложно получить |

HRP не требует ни прогноза доходностей, ни обращения матрицы. Он работает только с **матрицей корреляций**, что делает его устойчивым и воспроизводимым.

## 2. Математические основы

### 2.1 Логарифмические доходности

Для каждого актива $i$ рассчитывается временной ряд логарифмических доходностей:

$$r_{i,t} = \ln\!\left(\frac{P_{i,t}}{P_{i,t-1}}\right)$$

где $P_{i,t}$ — цена закрытия в момент $t$.

Логарифмические доходности предпочтительны перед простыми $\left(\frac{P_t - P_{t-1}}{P_{t-1}}\right)$, так как:
- они аддитивны во времени;
- лучше соответствуют нормальному распределению;
- симметричны (рост на 10% и падение на 10% — одинаковый масштаб).

### 2.2 Матрица корреляций

По логарифмическим доходностям рассчитывается матрица корреляций Пирсона $\mathbf{C}$:

$$\rho_{ij} = \frac{\text{cov}(r_i, r_j)}{\sigma_i \sigma_j}$$

$$\mathbf{C} = \begin{pmatrix}
1 & \rho_{12} & \cdots & \rho_{1n} \\
\rho_{21} & 1 & \cdots & \rho_{2n} \\
\vdots & \vdots & \ddots & \vdots \\
\rho_{n1} & \rho_{n2} & \cdots & 1
\end{pmatrix}$$

**Пример** для четырёх активов (SBERP, LKOH, BTC-USD, USDT-USD):

```
         SBERP   LKOH   BTC   USDT
SBERP  [ 1.00   0.72  -0.03  0.01 ]
LKOH   [ 0.72   1.00  -0.01  0.00 ]
BTC    [-0.03  -0.01   1.00  0.05 ]
USDT   [ 0.01   0.00   0.05  1.00 ]
```

Акции нефтяного сектора (SBERP и LKOH) сильно коррелируют между собой (0.72), тогда как биткоин и стейблкоин практически некоррелированы с ними.

### 2.3 Преобразование корреляции в расстояние

Для работы алгоритма кластеризации корреляция преобразуется в метрику расстояния. Формула López de Prado:

$$d_{ij} = \sqrt{\frac{1 - \rho_{ij}}{2}}$$

Свойства этой метрики:
- $d_{ij} \in [0, 1]$
- $d_{ij} = 0$ при $\rho_{ij} = 1$ (идеальная прямая корреляция)
- $d_{ij} = 1$ при $\rho_{ij} = -1$ (идеальная обратная корреляция)
- $d_{ij} = \frac{1}{\sqrt{2}} \approx 0.707$ при $\rho_{ij} = 0$ (нет корреляции)

В C# это выглядит так:

```csharp
double Distance(double rho) => Math.Sqrt(0.5 * (1 - rho));
```

### 2.4 Иерархическая кластеризация

На матрице расстояний запускается алгоритм **single-linkage hierarchical clustering** (метод ближайшего соседа):

1. Каждый актив — отдельный кластер.
2. Находим два кластера с минимальным расстоянием между ними.
3. Объединяем их в один кластер.
4. Повторяем, пока не останется один кластер.

Расстояние между кластерами (single linkage):

$$D(A, B) = \min_{i \in A,\, j \in B} d_{ij}$$

Результат — **дендрограмма** (дерево кластеров):

```
                     ┌── SBERP
              ┌──────┤
              │      └── LKOH
──────────────┤
              │      ┌── BTC-USD
              └──────┤
                     │      ┌── ETH-USD
                     └──────┤
                            └── TAO-USD
```

### 2.5 Квазидиагонализация

Порядок листьев дендрограммы задаёт **переупорядочивание активов** так, чтобы похожие (сильно коррелированные) активы стояли рядом. После этой перестановки матрица корреляций приобретает блочно-диагональный вид — «квазидиагональную» структуру.

**До** квазидиагонализации (случайный порядок):

```
BTC  | SBERP | USDT | LKOH | ETH
-----+-------+------+------+----
1.00 |  0.00 | 0.05 | 0.00 | 0.85  BTC
0.00 |  1.00 | 0.01 | 0.72 | 0.00  SBERP
0.05 |  0.01 | 1.00 | 0.00 | 0.04  USDT
0.00 |  0.72 | 0.00 | 1.00 | 0.00  LKOH
0.85 |  0.00 | 0.04 | 0.00 | 1.00  ETH
```

**После** квазидиагонализации (похожие активы рядом):

```
SBERP | LKOH | USDT | BTC  | ETH
------+------+------+------+----
1.00  | 0.72 | 0.01 | 0.00 | 0.00  SBERP
0.72  | 1.00 | 0.00 | 0.00 | 0.00  LKOH
0.01  | 0.00 | 1.00 | 0.05 | 0.04  USDT
0.00  | 0.00 | 0.05 | 1.00 | 0.85  BTC
0.00  | 0.00 | 0.04 | 0.85 | 1.00  ETH
```

### 2.6 Рекурсивное распределение весов

Теперь применяется **recursive bisection** (рекурсивное бисекционное деление):

1. Весь список активов (в квазидиагональном порядке) делится пополам.
2. Для каждой половины вычисляется **кластерная дисперсия** как обратная взвешенная сумма дисперсий:
   $$\tilde{\sigma}_C^2 = \frac{1}{\sum_{i \in C} \sigma_i^{-2}}$$
3. Вес каждой половины пропорционален **обратной дисперсии** (inverse variance):
   $$\alpha = \frac{1/\tilde{\sigma}_L^2}{1/\tilde{\sigma}_L^2 + 1/\tilde{\sigma}_R^2}$$
4. Рекурсия продолжается, пока каждый актив не получит свой вес.

**Схема рекурсии** (пример с 4 активами):

```
[SBERP, LKOH, BTC, ETH]       sigma2_left=0.0012, sigma2_right=0.0250
     left: alpha_L=0.954     right: alpha_R=0.046

[SBERP, LKOH]               [BTC, ETH]
sigma2_left=0.0010           sigma2_left=0.0230
sigma2_right=0.0015          sigma2_right=0.0280

SBERP: 0.954 x 0.60 = 0.572   BTC: 0.046 x 0.55 = 0.025
LKOH:  0.954 x 0.40 = 0.382   ETH: 0.046 x 0.45 = 0.021
```

Итоговые веса: SBERP=57.2%, LKOH=38.2%, BTC=2.5%, ETH=2.1%  
(сумма = 100%)

## 5. Как добавить новый тикер

### Шаг 1. Создайте CSV-файл

Создайте файл `Data/SampleQuotes/{TICKER}.csv` с историческими ценами закрытия:

```
Date,Close
2024-01-10,150.00
2024-01-17,152.50
2024-01-24,149.75
2024-01-31,155.20
2024-02-07,158.00
...
```

**Требования к формату:**
- Разделитель — запятая (`,`)
- Дата в формате `YYYY-MM-DD`
- Цена закрытия — десятичное число (разделитель — точка)
- Заголовок строго `Date,Close`
- Кодировка UTF-8 (важно для русскоязычных тикеров типа `ПАРУС-ЛОГ`)
- Минимум **10–20 строк** для корректного расчёта корреляций
- Рекомендуется **60+ строк** (60 торговых дней = ~3 месяца)

### Шаг 2. Путь к файлу

```
Data/
└── SampleQuotes/
    ├── SBERP.csv          <- уже существует
    ├── NEWTICKET.csv      <- ваш новый файл
    └── ...
```

### Шаг 3. Обновите набор активов в коде

Если вы хотите, чтобы новый тикер участвовал в HRP-расчёте, добавьте его в список активов:

```csharp
// Пример настройки набора активов
var universe = new[]
{
    new Asset("SBERP",      "Сбербанк России",      AssetClass.Equity),
    new Asset("LKOH",       "НК ЛУКОЙЛ",            AssetClass.Equity),
    new Asset("BTC-USD",    "Bitcoin",              AssetClass.Crypto),
    // Добавьте ваш тикер:
    new Asset("NEWTICKET",  "Новый актив",          AssetClass.Equity),
};
```

### Шаг 4. Пример конфигурации через словарь

Для удобства можно вынести список тикеров в `appsettings.json` или константы:

```json
{
  "Portfolio": {
    "Tickers": [
      "SBERP", "ZAYM", "SIBN", "BELU", "TATNP",
      "PLZL", "DOMRF", "PHOR", "MOEX", "MDMG",
      "LKOH", "ПАРУС-ЛОГ", "XXXXXX",
      "BTC-USD", "TAO-USD", "ETH-USD", "USDT-USD",
      "NEWTICKET"
    ]
  }
}
```

### Шаг 5. Проверка

Запустите тесты — дымовой тест `Load_RealSampleData_AllTickersLoadSuccessfully` автоматически загружает все файлы из папки `Data/SampleQuotes/`:

```bash
dotnet test AssetRebalancing.Tests/AssetRebalancing.Tests.csproj
```

## 7. Практические рекомендации

### Окно расчёта корреляций

| Окно | Характеристика |
|---|---|
| 20–30 дней | Очень реактивное, нестабильное |
| **60 дней** | Хороший баланс (рекомендуется как минимум) |
| **120 дней** | Более устойчивое, медленнее реагирует |
| 252 дня (1 год) | Долгосрочная перспектива для проверки |

**Рекомендация:** рассчитывайте корреляции на окне 60 дней, но проверяйте на 252 днях как «санити-чек».

### Пороги ребалансировки

`ThresholdRebalancer` по умолчанию использует правило **5/20**:
- Абсолютное отклонение >= **5 п.п.** (например, целевой вес 15%, текущий 10%)
- Относительное отклонение >= **20%** от целевого (например, цель 15%, факт 12% = отклонение 20%)

Это правило минимизирует транзакционные издержки при сохранении качества ребалансировки.

### Частота обновления котировок

Файлы в `Data/SampleQuotes/` обновляются вручную. Рекомендуемая частота:
- **Еженедельно** — для большинства портфелей
- **Ежемесячно** — если ребалансировка по расписанию

### Рекомендации по диверсификации

Типичные «строительные блоки» портфеля с низкой взаимной корреляцией:

| Блок | Примеры тикеров | Функция в портфеле |
|---|---|---|
| Акции нефтяного сектора | SIBN, LKOH, TATNP | Рост при высокой нефти |
| Акции финансового сектора | SBERP, MOEX | Широкий рост экономики |
| Сырьё / золото | PLZL | Хедж от инфляции и геополитики |
| Недвижимость (ЗПИФ) | ПАРУС-ЛОГ, XXXXXX | Стабильный рентный доход |
| Криптовалюта | BTC-USD, ETH-USD | Рискованный высокодоходный актив |
| Стейблкоин / кэш | USDT-USD, DOMRF | Стабилизатор, хранение ликвидности |

### Ledoit-Wolf shrinkage

При большом числе активов (> 15) «сырая» матрица корреляций становится нестабильной. Рекомендуется применять **Ledoit-Wolf shrinkage** для регуляризации — это уменьшает шум в оценке корреляций. В `MathNet.Numerics` можно реализовать за ~20 строк кода.

### Walk-forward бэктест

Для оценки качества стратегии рекомендуется **walk-forward** тестирование:
1. Взять скользящее окно (например, 120 дней).
2. Рассчитать HRP-веса на этом окне.
3. «Держать» портфель следующие 20 дней (out-of-sample).
4. Сдвинуть окно на 20 дней вперёд, повторить.
5. Оценить Sharpe Ratio, Max Drawdown, Calmar Ratio.
